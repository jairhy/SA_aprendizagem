1. O que é um SGBD? Tipos e Aplicações

O que é SGBD?

SGBD (Sistema Gerenciador de Banco de Dados) é um software que permite criar, gerenciar, organizar e manipular bancos de dados. Ele funciona como uma interface entre os usuários/aplicações e os dados armazenados, garantindo segurança, integridade e eficiência no acesso às informações.

Principais Funções de um SGBD:

Armazenamento: Guarda dados de forma estruturada

Consulta: Permite buscar informações rapidamente

Atualização: Modifica dados existentes

Segurança: Controla acesso e protege informações

Integridade: Garante consistência dos dados

Backup: Realiza cópias de segurança

Concorrência: Permite múltiplos acessos simultâneos

Tipos de SGBD:

1. SGBD Relacional (RDBMS)

  • Organiza dados em tabelas com linhas e colunas
  • Usa SQL (Structured Query Language)
  • Exemplos: MySQL, PostgreSQL, Oracle, SQL Server
  • Ideal para: sistemas transacionais, ERPs, sistemas bancários

    • 2. SGBD NoSQL

    Não usa tabelas relacionais tradicionais

    Mais flexível e escalável

    Tipos: Documentos (MongoDB), Chave-Valor (Redis), Colunas (Cassandra), Grafos (Neo4j)

    Ideal para: big data, redes sociais, IoT, aplicações em tempo real

    3. SGBD Hierárquico

    Organiza dados em estrutura de árvore

    Relacionamento pai-filho

    Exemplo: IBM IMS

    Ideal para: sistemas legados, estruturas organizacionais

    4. SGBD de Rede

    Permite múltiplos relacionamentos entre dados

    Mais flexível que o hierárquico

    Exemplo: IDMS

    Aplicações de SGBD:

    Sistemas Bancários: Gerenciamento de contas, transações

    E-commerce: Catálogo de produtos, pedidos, clientes

    Redes Sociais: Perfis de usuários, posts, conexões

    Saúde: Prontuários eletrônicos, agendamentos

    Educação: Sistemas acadêmicos, notas, matrículas

    Recursos Humanos: Folha de pagamento, cadastro de funcionários

    Logística: Controle de estoque, rastreamento

    3. O que é um Dicionário de Dados? Quais os Tipos de Dados?

    Dicionário de Dados

    O Dicionário de Dados é um repositório centralizado que contém informações sobre a estrutura, organização e definições de todos os dados armazenados no banco de dados. Funciona como um "manual" que documenta cada elemento do banco.

    Informações Contidas no Dicionário:

    Nome das tabelas: Identificação de cada tabela

    Nome das colunas: Campos de cada tabela

    Tipo de dados: Tipo de informação armazenada

    Tamanho: Limite de caracteres ou bytes

    Restrições: NOT NULL, UNIQUE, PRIMARY KEY, FOREIGN KEY

    Descrição: Explicação do propósito do campo

    Relacionamentos: Como as tabelas se conectam

    Índices: Estruturas para acelerar buscas

    Tipos de Dados em Bancos de Dados:

    1. Numéricos

    INT / INTEGER: Números inteiros (ex: 1, 100, -50)

    SMALLINT: Inteiros pequenos (-32768 a 32767)

    BIGINT: Inteiros grandes

    DECIMAL / NUMERIC: Números com precisão decimal fixa (ex: preços)

    FLOAT / REAL: Números com ponto flutuante (aproximados)

    DOUBLE: Ponto flutuante com precisão dupla

    div

    CHAR(n): Texto com tamanho fixo (ex: UF com 2 caracteres)

    VARCHAR(n): Texto com tamanho variável (ex: nome, endereço)

    TEXT: Texto longo sem limite específico

    LONGTEXT: Texto muito longo (até 4GB)

    3. Data e Hora

    DATE: Data (AAAA-MM-DD)

    TIME: Hora (HH:MM:SS)

    DATETIME: Data e hora combinados

    TIMESTAMP: Data/hora com fuso horário

    YEAR: Apenas o ano

    4. Booleano / Lógico

    BOOLEAN / BOOL: Verdadeiro (TRUE/1) ou Falso (FALSE/0)

    BIT: Valores binários (0 ou 1)

    5. Binários

    BLOB: Binary Large Object (imagens, arquivos)

    BINARY: Dados binários de tamanho fixo

    VARBINARY: Dados binários de tamanho variável

    6. Especiais

    JSON: Documentos JSON estruturados

    XML: Documentos XML

    ENUM: Lista de valores permitidos

    UUID: Identificador único universal

    4. O que é Modelagem de Banco de Dados? Modelo DER e MER

    Modelagem de Banco de Dados

    A Modelagem de Banco de Dados é o processo de criar uma representação visual e conceitual da estrutura de dados que será armazenada no banco. É como fazer uma "planta arquitetônica" antes de construir um banco de dados, definindo tabelas, campos, relacionamentos e regras.

    Níveis de Modelagem:

    Conceitual: Visão de alto nível, independente de tecnologia (MER)

    Lógico: Define estruturas sem detalhes de implementação (DER)

    Físico: Implementação específica do SGBD (SQL, índices, partições)

    MER - Modelo Entidade-Relacionamento

    O MER (Modelo Entidade-Relacionamento) é uma representação conceitual de alto nível que descreve os dados e suas relações sem se preocupar com detalhes técnicos de implementação.

    Componentes do MER:

    Entidades: Objetos do mundo real (Cliente, Produto, Pedido)

    Atributos: Características das entidades (nome, preço, data)

    Relacionamentos: Como as entidades se conectam (Cliente FAZ Pedido)

    Cardinalidade: Quantidade de ocorrências (1:1, 1:N, N:M)

    DER - Diagrama Entidade-Relacionamento

    O DER (Diagrama Entidade-Relacionamento) é a representação gráfica do MER. É o desenho visual que mostra entidades, atributos e relacionamentos usando símbolos padronizados.

    Símbolos do DER:

    Retângulo: Representa entidades (tabelas)

    Elipse/Oval: Representa atributos (campos)

    Losango: Representa relacionamentos

    Linhas: Conectam entidades aos relacionamentos

    Sublinhado: Indica chave primária

    Tipos de Relacionamentos:

    1:1 (Um para Um): Pessoa ↔ CPF

    1:N (Um para Muitos): Cliente → Pedidos

    N:M (Muitos para Muitos): Alunos ↔ Disciplinas

    Diferença entre MER e DER:

    MER: É o MODELO (conceito teórico, abstrato)

    DER: É o DIAGRAMA (representação visual do MER)

    Em resumo: MER é a teoria, DER é o desenho dessa teoria.

    5. Exemplo de Diagrama DER

    Sistema de Biblioteca - Diagrama DER

    Vamos analisar um diagrama DER para um sistema de biblioteca que gerencia livros, autores e empréstimos.

    Explicação do Diagrama DER:

    Entidades (Retângulos):

    LIVRO: Representa os livros da biblioteca

    id_livro (PK - Chave Primária)

    titulo

    isbn

    ano_publicacao

    editora

    AUTOR: Representa os autores dos livros

    id_autor (PK)

    nome

    nacionalidade

    data_nascimento

    div

    USUARIO: Representa os usuários da biblioteca

    id_usuario (PK)

    nome

    cpf

    email

    telefone

    EMPRESTIMO: Representa os empréstimos realizados

    id_emprestimo (PK)

    data_emprestimo

    data_devolucao_prevista

    data_devolucao_real

    status

    Relacionamentos (Losangos):

    ESCREVE: Relacionamento N:M entre AUTOR e LIVRO

    Um autor pode escrever vários livros

    Um livro pode ter vários autores

    REALIZA: Relacionamento 1:diventre USUARIO e EMPRESTIMO

    Um usuário pode realizar vários empréstimos

    Um empréstimo é feito por apenas um usuário

    CONTÉM: Relacionamento 1:N entre EMPRESTIMO e LIVRO

    Um empréstimo pode conter vários livros

    Cada livro em um empréstimo pertence a apenas um empréstimo específico

    Cardinalidades:

    N:M (Muitos para Muitos): AUTOR ↔ LIVRO (precisa de tabela intermediária)

    1:N (Um para Muitos): USUARIO → EMPRESTIMO

    1:N (Um para Muitos): EMPRESTIMO → LIVRO

    6. explicação extre sobre Diagrama MER

    Sistema de E-commerce - Modelo Entidade-Relacionamento

    Vamos analisar um MER para um sistema de e-commerce que gerencia clientes, produtos e pedidos.

    Explicação do Modelo MER:

    Entidades:

    CLIENTE

    Atributos: id_cliente, nome, email, senha, cpf, data_cadastro

    Chave Primária: id_cliente

    ENDERECO

    Atributos: id_endereco, rua, numero, complemento, bairro, cidade, estado, cep

    Chave Primária: id_endereco

    PRODUTO

    Atributos: id_produto, nome, descricao, preco, estoque, categoria

    Chave Primária: id_produto

    PEDIDO

    Atributos: id_pedido, data_pedido, valor_total, status, forma_pagamento

    Chave Primária: id_pedido

    ITEM_PEDIDO

    Atributos: id_item, quantidade, preco_unitario, subtotal

    Chave Primária: id_item

    Relacionamentos:

    POSSUI (1:N): CLIENTE possui múltiplos ENDERECOS

    Um cliente pode ter vários endereços cadastrados

    Cada endereço pertence a apenas um cliente

    FAZ (1:N): CLIENTE faz múltiplos PEDIDOS

    Um cliente pode fazer vários pedidos

    Cada pedido é feito por apenas um cliente

    CONTEM (N:M): PEDIDO contém múltiplos PRODUTOS

    Um pedido pode ter vários produtos

    Um produto pode estar em vários pedidos

    Resolvido pela entidade associativa ITEM_PEDIDO

    Regras de Negócio:

    Todo pedido deve ter pelo menos um item

    O valor total do pedido é calculado pela soma dos subtotais dos itens

    O estoque do produto deve ser atualizado quando um pedido é confirmado

    Um cliente deve ter pelo menos um endereço cadastrado

    O status do pedido pode ser: Pendente, Confirmado, Em Transporte, Entregue, Cancelado

    Diferença Visual MER vs DER:

    MER: Mais textual, focado na descrição conceitual

    DER: Mais gráfico, usa símbolos padronizados (retângulos, losangos, elipses)
